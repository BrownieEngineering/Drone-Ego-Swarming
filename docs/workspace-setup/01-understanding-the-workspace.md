# Chapter 3.1 — Understanding the Workspace

**Estimated Time:** 10–15 minutes

---

# Overview

Before building the Drone EGO Swarming project, it is important to
understand how a ROS 2 workspace is organized.

Unlike many software projects that consist of a single repository,
ROS 2 projects are commonly built from multiple packages inside a
workspace.

The Drone EGO Swarming project follows this standard ROS 2 structure.

Understanding the purpose of each workspace directory will make future
builds, debugging, and package management a little easier

---

# What is a ROS 2 Workspace?

A ROS 2 workspace is a directory that contains one or more ROS packages
which are built together using **colcon**.

The workspace provides a single environment where packages can:

- communicate with one another,
- share message definitions,
- depend on one another,
- and be launched together.

Every package used by the Drone EGO Swarming project is ultimately built
inside the workspace.

---

# Workspace Layout

The workspace used throughout this manual is:

```text
~/ego_ws
```

After building the workspace, the directory will look similar to:

```text
ego_ws/

├── src/
├── build/
├── install/
└── log/
```

Each directory serves a different purpose.

---

# The src Directory

```text
~/ego_ws/src
```

The **src** directory stores the source code for every ROS package that
will be built.

Examples would be:

- EGO Planner
- px4_msgs
- px4_ego
- additional ROS packages

Only source code belongs inside this directory.

Whenever a new ROS package is added to the project, it should be placed
inside:

```text
~/ego_ws/src
```

---

# The build Directory

```text
~/ego_ws/build
```

The **build** directory is automatically created by **colcon**.

It stores temporary compilation files used while building packages.

Normally, this directory should never be edited manually.

If necessary, it can safely be deleted before rebuilding the workspace.

---

# The install Directory

```text
~/ego_ws/install
```

The **install** directory contains the compiled ROS packages that are
used while running the project.

When you execute:

```bash
source ~/ego_ws/install/setup.bash
```

ROS loads the packages from this directory.

This is why the install directory is sourced instead of the src or build
directories.

---

# The log Directory

```text
~/ego_ws/log
```

The **log** directory stores build logs generated by **colcon**.

If a package fails to build, the logs in this directory can help identify
the problem.

Like the build directory, it is generated automatically.

---

# How Packages Become Available

Building the workspace does not automatically make packages available.

The general flow is:

```text
Source Code

↓

colcon build

↓

install/

↓

source install/setup.bash

↓

ROS can now use the packages
```

This process will be repeated many times throughout development.

---

# Common Workspace Commands

Move into the workspace:

```bash
cd ~/ego_ws
```

Move into the source directory:

```bash
cd ~/ego_ws/src
```

Build every package:

```bash
colcon build
```

Build a single package:

```bash
colcon build --packages-select <package_name>
```

Source the workspace:

```bash
source ~/ego_ws/install/setup.bash
```

# Common Issues

---

## Package not found

Cause

The package has not been built or the workspace has not been sourced.

Solution

Build the workspace:

```bash
colcon build
```

Then source it:

```bash
source ~/ego_ws/install/setup.bash
```

---

## Changes are not appearing

Cause

The workspace has not been rebuilt.

Solution

Run:

```bash
colcon build
```

and restart the affected ROS node.

---

## Old code is still running

Cause

The workspace was rebuilt, but the running node was not restarted.

Solution

Stop the running node.

Rebuild if necessary.

Source the workspace.

Launch the node again.
