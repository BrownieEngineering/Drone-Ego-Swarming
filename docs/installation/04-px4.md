# Chapter 2.4 — PX4 Autopilot

**Estimated Time:** 45–90 minutes

---

# Overview

PX4 Autopilot is the flight control software used throughout the Drone
EGO Swarming project.

It is responsible for controlling the simulated aircraft, maintaining
vehicle stability, receiving trajectory commands, estimating vehicle
state, and executing autonomous flight.

Within this project, PX4 operates in Software-In-The-Loop (SITL) mode,
meaning that the complete flight controller runs as software while
Gazebo simulates the physical drone.

Unlike Gazebo, which simulates the environment, PX4 simulates the
actual flight controller.

---

# What is PX4?

PX4 is an open-source flight control stack developed specifically for
autonomous aerial vehicles.

PX4 is responsible for:

- vehicle arming
- flight modes
- state estimation
- attitude control
- position control
- velocity control
- failsafes
- trajectory following

It supports:

- quadcopters
- fixed-wing aircraft
- VTOL aircraft
- ground vehicles
- boats

The Drone EGO Swarming project currently uses the multicopter
configuration.

---

# Why PX4?

PX4 was selected because it provides:

- native ROS 2 support
- Micro XRCE-DDS communication
- Offboard trajectory control
- active community support
- excellent Gazebo integration
- extensive documentation
- compatibility with EGO Planner

Most importantly, PX4 exposes trajectory interfaces that allow EGO
Planner to command autonomous flight.

---

# Why This Version?

The Drone EGO Swarming project was developed using:

```text
Commit:
25b7d627da

Description:
v1.18.0-alpha1-469-g25b7d627da
```

This version is important because PX4 ROS topics are versioned.

For example:

```text
/fmu/out/vehicle_status_v4
```

Earlier versions may instead publish:

```text
/fmu/out/vehicle_status

or

/fmu/out/vehicle_status_v1
```

This difference became one of the largest debugging challenges during
development.

Using the tested commit eliminates this compatibility issue.

---

# How PX4 Fits Into Drone EGO Swarming

PX4 acts as the flight controller inside the complete software stack.

```text
RViz

↓

Goal Pose

↓

EGO Planner

↓

PositionCommand

↓

PX4 Bridge

↓

TrajectorySetpoint

↓

PX4

↓

Gazebo Vehicle
```

PX4 receives trajectory commands generated by EGO Planner and converts
them into simulated flight.

# Clone PX4

Navigate to your home directory.

```bash
cd ~
```

Clone PX4 with all required submodules.

```bash
git clone https://github.com/PX4/PX4-Autopilot.git --recursive
```

Enter the repository.

```bash
cd ~/PX4-Autopilot
```

---

# Checkout the Tested Version

Switch to the tested commit.

```bash
git checkout 25b7d627da
```

Update every submodule.

```bash
git submodule update --init --recursive
```

This guarantees that every dependency matches the tested project.

---

# Install PX4 Dependencies

Run the official Ubuntu setup script.

```bash
bash ./Tools/setup/ubuntu.sh
```

This installs the dependencies required by PX4.

The script may request that you restart your terminal.

---

# Build PX4

Compile the Software-In-The-Loop version.

```bash
make px4_sitl_default
```

This process may take several minutes during the first build.

---

Verify the binary exists.

# Common Issues

---

## Submodule errors

Cause

The repository was cloned without submodules.

Solution

Run:

```bash
git submodule update --init --recursive
```

---

## Build failed

Most build failures occur because dependencies were not installed.

Run:

```bash
bash ./Tools/setup/ubuntu.sh
```

Restart the terminal.

Rebuild.

---

## Wrong commit

Verify:

```bash
git rev-parse --short HEAD
```

If the commit does not match:

```text
25b7d627da
```

switch to the tested commit.

---

## Wrong branch

Verify:

```bash
git branch
```

Expected:

```text
* main
```

after checking out the tested commit.

---

## PX4 binary missing

Verify:

```bash
ls build/px4_sitl_default/bin/
```

If no binary exists:

```bash
make px4_sitl_default
```

again.
